# Poetry 仮想環境の管理と同期時の注意

このディレクトリは Dropbox 等を介して複数台の計算機で同期されますが、Poetry が作成するローカル仮想環境の置き場所（`.cache` 以下など）は計算機ごとに異なります。
このドキュメントでは、別の計算機で環境を再現する際の正しい手順とその理由を説明します。

## 結論: 別の計算機で作業を再開する手順

同期元の計算機で `poetry.lock` を更新・同期済みである場合、別の計算機で作業を再開する際には **再ロック（`poetry lock`）は不要** です。以下の2ステップのみを実行してください。

```bash
# 1. その計算機にインストールされている Python 3.12 を新規の仮想環境に紐付ける
poetry env use python3.12

# 2. 同期された poetry.lock の構成通りにパッケージをインストールする
poetry install
```

## よくある誤解と解説

### `poetry.lock` にローカル環境のパスは書かれていない
`poetry.lock` に記録されているのは、「このプロジェクトを動作させるのに必要な全パッケージの**正確なバージョン番号**と**ハッシュ値（改ざん防止のデータ）**」だけです。
仮想環境が作成された各自のローカルパス情報などは **記載されていません**。これは「どの計算機でも全く同じバージョンのライブラリを入れて動作を再現できるようにするため」の仕様です。Dropbox で同期されても環境そのものが壊れるわけではありません。

### `poetry lock` をやり直してはいけない理由
計算機が変わって環境が認識できなくなったからといって、慌てて `poetry lock` を実行してしまうと、以下のような問題が発生します。

- 同期元との間で **ファイルのコンフリクト（競合）** が発生する。
- 意図せずパッケージの最新版を取得し直してしまい、**環境の再現性が失われる**（予期せぬアップデートによるバグが生じるリスクがある）。

`poetry.lock` は手動で更新せずそのまま同期させ、各計算機の Poetry に `poetry install` をお願いして「ロックファイルに記載された通りの環境」をそれぞれの計算機内部に再構築させるのが正しい運用方法です。
